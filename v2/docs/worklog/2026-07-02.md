## Summary

Completed the first `azd up` of CWYD v2 on the new tenant/subscription. Provisioning + build/push/deploy of all three container services **succeeded**, and the post-provision index + knowledge base seeded. But a live review of the running deployment found the **backend Container App crashes on startup**: it reads a Cosmos DB container during the FastAPI lifespan and gets `(Forbidden) ... blocked by your Cosmos DB account firewall`, so `/api/health` never responds. Root cause is that Cosmos DB + Storage come up with `publicNetworkAccess: Disabled` in the default public profile — logged as [BUG-0093](../bugs.md).

## Planned

- Repoint the azd env to the new subscription, verify quota, run the first `azd up`.
- Review the live deployment (health + grounding).

## Done

- Repointed the azd env to the new dev subscription (cleared the stale old-tenant subscription/tenant); confirmed `azd auth login` works for the new account (`az login` is not available on it, so the deploy is azd-driven).
- Verified model quota in `eastus2`: `gpt-5.1` GlobalStandard 0/1000, `text-embedding-3-large` Standard 0/350 — ample. Subscription state **Enabled**.
- `azd provision --preview` validated the binding; then `azd up` (run from the **v2** sync terminal — a fresh async terminal picks up the **root v1** `azure.yaml` and fails; the `--cwd` flag did **not** redirect azd, so the project must be the shell cwd).
- **`azd up` succeeded:** provisioned Log Analytics, ACR, App Insights, Speech, Storage, Foundry + project, **gpt-5.1** + embedding deployments, Content Safety, Container Apps Environment, Cosmos DB, Search, the Foundry↔Search connection, and all three Container Apps (`ca-backend`, `ca-frontend`, `ca-func`). Post-provision created the `cwyd-index` search index + `cwyd-kb` knowledge base + KB-MCP connection. Endpoints emitted.
- **Live review found the backend down:** `/api/health` times out (0 bytes, 90s). Diagnosed via `az containerapp logs`: uvicorn starts, hangs at "Waiting for application startup", then `CosmosHttpResponseError: (Forbidden) ... blocked by your Cosmos DB account firewall` → `Application startup failed. Exiting.`. Confirmed the deployed Cosmos + Storage have `publicNetworkAccess: Disabled` (Search + AI Services are `Enabled`) despite the public profile. → [BUG-0093](../bugs.md).
- **Hotfix + recovery:** set Cosmos + Storage `publicNetworkAccess` to `Enabled`, restarted the backend revision → `/api/health` + `/api/health/ready` return `200 {"status":"pass"}` (foundry_iq / database / search all pass); frontend serves `200`.
- **Grounding validated:** re-ran the postdeploy seed non-interactively (`AZURE_ENV_SAMPLE_DATA=default`) — uploaded 6 benefits PDFs, index reached 15 docs (PASS). `POST /api/conversation` ("What does the Northwind Health Plus plan cover?") returned a grounded answer with a `[doc1]` citation to `Benefit_Options.pdf`. The deployment is functional end-to-end.

## Bugs

- [BUG-0093](../bugs.md) (**fixed**, blocker, infra) -- default public-profile `azd up` yields a non-functional backend: Cosmos DB + Storage come up `publicNetworkAccess: Disabled` (AVM modules ignore the `enablePrivateNetworking ? 'Disabled' : 'Enabled'` intent), so the public-egress backend can't reach Cosmos and crashes on startup. **Live hotfix applied 2026-07-02** (Cosmos + Storage -> `Enabled`, backend restarted -> health `200 pass`); **durable fix landed 2026-07-02** -- `post_provision.py` `_ensure_public_network_access` re-asserts `Enabled` on every provision (MACAE `selecting_team_config_and_data.sh` parity), verified live (`az` reports both `Enabled`); 7 unit tests + AST gates green.

## Decisions

- **DEP-03:** Killed the parked `azd up` process — it had finished provision + deploy and was blocked at the interactive postdeploy "sample-data" menu. The seed can be re-run non-interactively (`--set default` / `AZURE_ENV_SAMPLE_DATA=default`) after the backend is unblocked.
- **OBS-01:** The embedding deployment came up as **`text-embedding-3-small`** (1536-dim index), not `text-embedding-3-large` as the research assumed. It deployed fine and the index matches (1536); flagged for confirmation that this is the intended model in `main.parameters.json`.
