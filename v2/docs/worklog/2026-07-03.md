### BUG-0061 ‚Äî durable Event Grid MI fix

- **Root cause (confirmed):** RBAC replication latency, not creation ordering. `dependsOn` orders resource *creation*, but Azure RBAC is eventually consistent, and the **system-assigned** topic MI does not exist until the AVM `system-topic` module completes ‚Äî so the `Storage Queue Data Message Sender` grant was authored moments before Event Grid's **synchronous** delivery-authorization preflight, with ~0 s replication lead ‚Üí 401.
- **MACAE comparison:** MACAE uses no Event Grid at all; it handles grant-then-use via idempotent azd `postprovision` `az` scripts (post-ARM, replication settled) + a documented replication-delay retry. No pattern to copy directly.
- **Fix (durable, mirrors the existing-topic reuse path):** deliver as the **shared UAMI** (`id-<SUFFIX>`, exists at t=0) instead of the topic's system MI. Three coordinated edits in [../infra/main.bicep](../infra/main.bicep): (a) AVM `system-topic` `managedIdentities.userAssignedResourceIds: [ userAssignedIdentity.outputs.resourceId ]`; (b) `eventGridQueueSenderRole` grants the role to the UAMI on the storage account with **no** dependency on the topic, so ARM creates it at the start of the deployment and it replicates through the multi-minute provisioning tail before the preflight runs; (c) `blobCreatedSubscription.deliveryWithResourceIdentity.identity` ‚Üí `UserAssigned`. `az bicep build` exit 0 (only pre-existing BCP081/BCP334 warnings elsewhere).
- **Rejected alternatives:** `deploymentScripts` grant-then-wait (needs `allowSharedKeyAccess=true` backing storage ‚Äî conflicts with ADR 0002); moving the subscription into a `postprovision` hook (kept as a backstop, unneeded once the grant is UAMI-based and topic-independent).

### F-5 fixed ‚Äî reuse-path Event Grid topic now carries the UAMI

- **Closed the last review finding the right way.** F-5 was a latent gap on the `useExistingEventGridTopic` reuse path: the reuse subscription delivers via `deliveryWithResourceIdentity` pointing at v2's UAMI, but the topic was referenced as `existing` (read-only) and so never assigned that identity ‚Äî Event Grid can only deliver as an identity the **topic** carries, so a v1-topic-reuse deployment would have failed EG's synchronous delivery preflight.
- **Fix:** converted `existingEventGridTopic` in [../infra/main.bicep](../infra/main.bicep) from an `existing` reference to a **managed** `Microsoft.EventGrid/systemTopics` resource ‚Äî an idempotent in-place PUT that leaves the immutable `source`/`topicType` unchanged (`source == effectiveStorageResourceId`, the reused storage) and adds the UAMI via a compile-time-resolvable `resourceId('Microsoft.ManagedIdentity/userAssignedIdentities', identityName)` key (a module runtime output can't be an identity dictionary key ‚Äî BCP120). ARM updates v1's topic in place, attaching the UAMI, so the reuse subscription's delivery identity resolves. Azure allows only one system topic per storage source, so re-declaring the same topic (same name + source) is a valid update, not a create-conflict.
- **Guard:** added `test_reuse_path_topic_carries_uami_identity` to [test_main_bicep.py](../../tests/infra/test_main_bicep.py) pinning that the reuse topic is managed (not `existing`) and carries the UAMI. `az bicep build` exit 0; `pytest tests/infra/test_main_bicep.py` ‚Üí 40 passed.
