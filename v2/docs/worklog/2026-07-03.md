### BUG-0061 — durable Event Grid MI fix

- **Root cause (confirmed):** RBAC replication latency, not creation ordering. `dependsOn` orders resource *creation*, but Azure RBAC is eventually consistent, and the **system-assigned** topic MI does not exist until the AVM `system-topic` module completes — so the `Storage Queue Data Message Sender` grant was authored moments before Event Grid's **synchronous** delivery-authorization preflight, with ~0 s replication lead → 401.
- **MACAE comparison:** MACAE uses no Event Grid at all; it handles grant-then-use via idempotent azd `postprovision` `az` scripts (post-ARM, replication settled) + a documented replication-delay retry. No pattern to copy directly.
- **Fix (durable, mirrors the existing-topic reuse path):** deliver as the **shared UAMI** (`id-<SUFFIX>`, exists at t=0) instead of the topic's system MI. Three coordinated edits in [../infra/main.bicep](../infra/main.bicep): (a) AVM `system-topic` `managedIdentities.userAssignedResourceIds: [ userAssignedIdentity.outputs.resourceId ]`; (b) `eventGridQueueSenderRole` grants the role to the UAMI on the storage account with **no** dependency on the topic, so ARM creates it at the start of the deployment and it replicates through the multi-minute provisioning tail before the preflight runs; (c) `blobCreatedSubscription.deliveryWithResourceIdentity.identity` → `UserAssigned`. `az bicep build` exit 0 (only pre-existing BCP081/BCP334 warnings elsewhere).
- **Rejected alternatives:** `deploymentScripts` grant-then-wait (needs `allowSharedKeyAccess=true` backing storage — conflicts with ADR 0002); moving the subscription into a `postprovision` hook (kept as a backstop, unneeded once the grant is UAMI-based and topic-independent).
